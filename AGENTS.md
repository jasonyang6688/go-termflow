# Repository Guidelines

## Project Structure & Module Organization

TermFlow is a Wails v2 desktop app with a Go backend and Vue frontend.

- `main.go`, `app.go`, and `wails.json` define the Wails application shell.
- `internal/ssh/` manages SSH sessions and terminal connections.
- `internal/store/` contains SQLite storage and migrations.
- `frontend/src/` contains Vue 3 source code: `components/`, `stores/`, `styles/`, and `assets/`.
- `frontend/wailsjs/` is generated by Wails; do not hand-edit generated bindings.
- `docs/` holds product notes, plans, and screenshots.
- `build/` contains platform build assets.

## Build, Test, and Development Commands

- `wails dev` runs the full app in live development mode with Vite hot reload.
- `wails build` creates a redistributable desktop package.
- `cd frontend && npm install` installs frontend dependencies used by Wails.
- `cd frontend && npm run dev` runs only the Vite frontend server.
- `cd frontend && npm run build` builds the frontend bundle.
- `go test ./...` runs Go package tests.

Use root-level Wails commands for integrated backend/frontend work. Use frontend commands when iterating on Vue-only changes.

## Coding Style & Naming Conventions

Go code must be `gofmt` formatted. Keep backend packages small and domain-focused under `internal/`, and prefer explicit error returns over hidden side effects.

Frontend code uses Vue single-file components with `<script setup lang="ts">`. Name components in `PascalCase.vue`, stores in `camelCase.ts`, and CSS files by purpose, such as `tokens.css` or `base.css`. Follow the existing two-space indentation in Vue templates and TypeScript. Reuse CSS variables from `frontend/src/styles/tokens.css` before adding new colors.

## Testing Guidelines

Add Go tests beside the package under test using `*_test.go` files and `TestXxx` function names. Run `go test ./...` before submitting backend changes.

No frontend test runner is currently configured. For frontend changes, at minimum run `cd frontend && npm run build`; add targeted tests if introducing a test framework or complex logic.

## Commit & Pull Request Guidelines

The current history uses Conventional Commit style, for example `feat: initial TermFlow Wails v2 app`. Prefer `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, and `chore:` prefixes.

Pull requests should include a concise summary, verification commands run, linked issues or planning docs when relevant, and screenshots for visible UI changes.

## Security & Configuration Tips

Do not commit credentials, private keys, connection secrets, local databases, or generated local runtime state. Keep SSH and storage changes conservative: validate inputs at boundaries, avoid logging secrets, and document any migration behavior in `internal/store/` changes.
